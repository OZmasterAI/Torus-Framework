  This is not the concurrent access / segfault issue the skip logic was designed for. That's a different problem. Here's what's actually happening:

  1. test_framework.py is not a standard pytest file. It runs test code at module level (top-level scope, outside any def test_*() functions). Pytest tries to collect the file by importing it, which executes all that top-level code.
  2. The skip logic at the top correctly detects the MCP server is running and sets MEMORY_SERVER_RUNNING = True. But the session_time regression test block (~line 1800) opens a second PersistentClient to the same ChromaDB database that the MCP server

  3. ChromaDB's Rust backend doesn't handle two processes accessing the same persistent database simultaneously. The first get_or_create_collection() calls succeed (they're lightweight metadata ops), but the actual .get() query at line 1806 triggers

  4. Because this code runs at module top-level during import, the exception crashes pytest's collection phase — meaning zero tests run, not even the ones that would pass fine.
