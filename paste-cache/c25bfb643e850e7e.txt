1. System Overview
AKIRA is a behavioral enforcement layer built around Claude Code that:

Prevents destructive actions before they execute (PreToolUse gates)
Tracks state changes after tool execution (PostToolUse tracker)
Persists institutional memory across sessions via a ChromaDB vector database (20,950+ memories)
Coordinates multi-agent workflows with agent teams, task dependencies, and quality gates
Self-heals on boot by resetting stale state and verifying system integrity
The core insight: an LLM agent forgets everything between sessions and makes the same mistakes repeatedly. AKIRA enforces discipline through code (hooks + gates) rather than relying on prompt instructions alone (which the LLM follows ~40% of the time without enforcement).

Key Statistics (from behavioral analysis)
40% of behaviors are truly learned by the LLM; 60% require gate enforcement
Correction rate: 23.5% (606 corrections out of 2,573 messages analyzed)
Top 3 mistake categories: breaking things (deploying without testing), not querying memory, not using parallel agents
Gate enforcement reduced recurring mistakes by >80% for covered categories

3. Hook System
Claude Code provides hook events — lifecycle points where external scripts can intercept, modify, or block tool execution. AKIRA registers Python scripts for all 13 available events.

3.1 Hook Events and Their Scripts
#    Event    Script    Timeout    Purpose    Can Block?
1    PreToolUse    enforcer.py --event PreToolUse    5000ms    Run all 13 gates before any tool executes    Yes (exit 2)
2    PostToolUse    enforcer.py --event PostToolUse    3000ms    Route to tracker.py for state tracking    No (fail-open)
3    PermissionRequest    auto_approve.py    3000ms    Auto-approve safe bash commands (SSH, git status)    No (approval only)
4    SessionStart    boot_v5.py    15000ms    Reset state, display dashboard, integrity checks    No
5    PreCompact    pre_compact.py    5000ms    Backup state files, inject context summary    No
6    SessionEnd    session_end.py    5000ms    Mark session ended, reset session_reset_done flag    No
7    SubagentStart    subagent_inject.py    3000ms    Inject context into subagent system prompt    No
8    PostToolUseFailure    tool_failure.py    3000ms    Log tool failures to failure_log.jsonl    No
9    UserPromptSubmit    prompt_guard.py    2000ms    Warn on deploy keywords, unknown IPs, frustration signals    No (warn-only)
10    SubagentStop    subagent_output_guard.py    2000ms    Validate subagent output (error phrases, anomalies)    No (warn-only)
11    Notification    notification_handler.py    1000ms    Log notifications    No
12    TeammateIdle    teammate_idle.py    3000ms    Quality gate for agent team members going idle    Yes (exit 2)
13    TaskCompleted    task_completed.py    3000ms    Quality gate for task completion (error checking)    Yes (exit 2)
3.2 Hook Communication Protocol
Claude Code → stdin (JSON) → Hook Script → exit code + stderr/stdout

Exit codes:
  0 = Allow (proceed with tool execution)
  2 = Block (stop tool execution, show stderr message to LLM)
  1 = Hook error (Claude Code treats as hook malfunction, not a block)

Messages:
  stderr → Displayed to the LLM as feedback
  stdout → Special: "approve" on stdout in PermissionRequest = auto-approve
           PreCompact stdout = injected into compacted context

4. Gate Enforcement Engine
The core of AKIRA's behavioral enforcement. Every tool call goes through up to 13 gates before execution.

4.1 Gate Contract
Every gate module exports a single function:

def check(tool_name: str, tool_input: dict, state: dict) -> tuple[bool, str]:
    """
    Args:
        tool_name: The tool being invoked (e.g., "Edit", "Bash", "Task")
        tool_input: The tool's parameters (e.g., {"file_path": "...", "command": "..."})
        state: Current enforcement state (read-only snapshot)

    Returns:
        (True, "")           — Allow the tool call
        (False, "[GATE N] BLOCKED: reason") — Block with explanation
    """
Critical rules:

Gates are read-only — they NEVER modify state.json (tracker handles mutations)
Gates are fail-closed — if a gate crashes with an exception, the tool call is BLOCKED (exit 2), not bypassed
The enforcer catches gate exceptions and converts them to blocks
4.2 Gate Execution Order
Gates run sequentially in a specific order. The first gate to return (False, message) blocks the tool call.

GATE_ORDER = [
    (17, gate_opus_model),        # 1. Model enforcement (fastest check)
    (24, gate_task_rate_limit),   # 2. Task spawn rate limiting (inline)
    (25, gate_no_powershell),     # 3. No PowerShell commands
    (14, gate_protected_files),   # 4. Two-tier file protection (cursed + critical)
    (22, gate_vps_freshness),     # 5. SSH verification before writing VPS status
    (10, gate_trading),           # 6. Trading files need Plan Mode + research
    (12, gate_akira_first),       # 7. Must query memory before building
    (13, gate_edit_lifecycle),    # 8. Edit → verify → save-to-memory pipeline
    (1,  gate_read_first),        # 9. Read before Edit/Write
    (6,  gate_no_destroy),        # 10. Block destructive bash commands
    (7,  gate_vps_sync),          # 11. Pull → backup → deep_query before deploy
    (11, gate_test_first),        # 12. Run tests before deploying to VPS
]
# Gate 18 (plan mode AKIRA save) is inline in enforcer.py = 13 total
4.3 Gate Details
Gate 17 — Model Enforcement
Purpose: All Task (subagent) spawns must use the highest-tier model. Blocks: Task calls with missing or non-opus model parameter. Why: Paid subscription means best model everywhere. Lower models produce worse results and waste time.

Gate 24 — Task Rate Limiting (Inline)
Purpose: Prevent runaway agent spawning. Blocks: More than 30 Task spawns within a 10-minute window. Warns at 25+. Mechanism: Reads task_spawns timestamps from state, prunes entries older than 10 minutes.

Gate 25 — No PowerShell
Purpose: Block all PowerShell usage in bash commands. Blocks: Any command containing PowerShell executables, cmdlets, or flags. Why: Antivirus software flags PowerShell one-liners as malicious. Bash handles everything needed. Patterns: powershell.exe, pwsh, Invoke-Expression, ConvertFrom-Json, -NoProfile, etc.

Gate 14 — Protected Files (Two-Tier)
Purpose: Extra protection for files that historically cause problems.

Tier 1 — CURSED files (60%+ frustration rate from behavioral analysis):

Files: trader.py, boot_v5.py, enforcer.py, active_bets.json, consensus*
Requires: Plan Mode + AKIRA query within 5 minutes
Cannot be bypassed even with plan approval
Tier 2 — CRITICAL files (system infrastructure):

Files: orchestrator, signal_detector, order_manager, risk_gate, monitor, etc.
Requires: AKIRA query specifically mentioning that file within 10 minutes
Skips when plan is approved (60-min TTL)
Gate 22 — VPS Freshness
Purpose: Prevent writing stale VPS data to status files. Blocks: Writes to HANDOFF.md or LIVE_STATE.json containing VPS status keywords (PID, RUNNING, PnL) unless an SSH command was run within 15 minutes. Why: Prevents the LLM from writing "everything is fine" from cached/hallucinated data.

Gate 10 — Trading = Plan Mode
Purpose: Trading code changes are high-risk (real money). Require extra process. Blocks: Edits to trading files without ALL of:

Active Plan Mode or recently approved plan
AKIRA query within 10 minutes
Research tool (deep_query or search_knowledge) — not just any AKIRA tool Trading file patterns: polymarket/, golden_v*, strategy.py, order_manager, signal_detector, trader_*.py
Gate 12 — AKIRA-First (+ Temporal Awareness)
Purpose: Must query institutional memory before making any code changes. Blocks: Edit/Write/NotebookEdit if AKIRA wasn't queried in the last 10 minutes. Temporal enhancement: During high-risk times (Friday, late night 11PM-3AM), block messages include the temporal risk reason for extra visibility. Exemptions: Reading tools, AKIRA tools themselves, plan files (HANDOFF.md, plans/)

Gate 13 — Edit Lifecycle (Two-Stage Pipeline)
Purpose: Enforce disciplined edit workflow: edit → verify → save knowledge.

Stage 1 — AKIRA Save Check (formerly Gate 19): After a fix is verified via tests/reading, ALL edits are blocked until the fix is saved to AKIRA memory. This prevents knowledge loss — the fix worked but was never recorded.

State field: fix_needs_akira_save (dict of {filename: True})
Cleared when: remember_this(), save_this(), or teach_pattern() is called
Stage 2 — Per-File Verification (formerly Gate 13): After editing a .py file, you must verify it works (read it back, run tests) before re-editing THE SAME file. Cross-file edits are allowed.

State field: pending_verification_files (list of normalized paths)
Cleared when: The specific file is Read back, or tests are run
Stage 0 — Workspace Isolation (optional): When agent teams are active, checks if a file is "claimed" by another agent. Prevents two agents from editing the same file simultaneously.

Both stages skip when plan_approved=True (with 60-minute TTL).

Gate 1 — Read Before Edit
Purpose: Never modify a file you haven't read first. Blocks: Edit/Write on existing files not in the files_read state list. Exemptions: .json, .lock, .txt, .log files; __pycache__, node_modules, .git directories; new files that don't exist yet.

Gate 6 — No Destructive Commands
Purpose: Block dangerous bash commands that could destroy data. Blocks: 44 regex patterns including:

File destruction: rm -rf, rm -r, find -delete, shred, truncate -s 0
SQL destruction: DROP TABLE, DELETE FROM (without WHERE)
Git destruction: git push --force, git reset --hard, git clean -f, git checkout .
System destruction: mkfs, dd if=, chmod 777 /
Service destruction: systemctl disable *trader*, pkill -9 *trader*
Pipe-to-shell: curl | bash, wget | bash, curl | python
Encoded execution: base64 | bash, eval "$var"
Interpreter execution: node -e, ruby -e, python -c (with SSH context) Smart skip: grep/rg commands are always allowed (searching FOR destructive patterns, not executing them).
Gate 7 — VPS Sync (Pull → Backup → Research)
Purpose: Three-gate compound check for VPS deployments. Blocks VPS deploy commands (scp/rsync TO VPS) if ANY of:

Gate 7: No VPS pull (scp FROM VPS) within 30 minutes
Gate 20: No backup (cp file file.bak) on VPS within 30 minutes
Gate 21: No deep_query about the component within 30 minutes
Gate 21b: The deep_query must mention the specific file being deployed
Gate 11 — Test Before Deploy
Purpose: Run tests before deploying anything to production VPS. Blocks: VPS deploy commands or SSH service restarts if tests_run_this_session is False. Test patterns: pytest, python -m pytest, py_compile, mypy, ruff check, generic python script.py.

Gate 18 — Plan Mode AKIRA Save (Inline in enforcer.py)
Purpose: Before exiting Plan Mode, save the plan to AKIRA memory. Blocks: ExitPlanMode if akira_saved_in_plan_mode is False and edits were made. Why: Plans represent significant thinking. Losing them is wasteful. Exception: If no edits were made during plan mode (empty pending_verification_files), plan abort is allowed without saving.

5. State Management
5.1 State File
All enforcement state is stored in a single JSON file (state.json) with atomic read-modify-write operations.

5.3 Atomic Operations
State management uses file locking to prevent concurrent access corruption:

def update_state(modifier_fn):
    """Atomic read-modify-write with file locking.

    1. Acquire lock file (state.json.lock)
    2. Read current state from state.json
    3. Call modifier_fn(state) to mutate in-place
    4. Write to temp file (state.json.tmp)
    5. Atomic rename (os.replace) temp → state.json
    6. Release lock

    If modifier_fn raises, state is NOT saved (prevents corruption).
    If lock acquisition fails after 5s, tries stale lock detection (>30s = break and retry).
    """
Cross-platform locking:

Windows: msvcrt.locking() with retry loop (50ms intervals, 5s timeout)
Unix: fcntl.flock() with equivalent retry loop
Fallback: No-op (if neither available)
Stale lock detection: If a hook process crashes while holding the lock, the lock file remains. After 5s timeout, the system checks if the lock file is >30 seconds old. If so, it breaks the lock and retries.

5.4 Session Reset
On each new session (detected by akira_last_queried being null or >1 hour old), all session-specific fields are reset to defaults. Only session_number persists across sessions.

def reset_session_state(state):
    """Reset per-session fields. Called once per new session."""
    state["files_read"] = []
    state["plan_mode"] = False
    state["plan_approved"] = False
    state["tests_run_this_session"] = False
    state["pending_verification_files"] = []
    state["akira_last_queried"] = None
    state["task_spawns"] = []
    # ... (all volatile fields reset)
    # KEEP: session_number (persistent)
6. PostToolUse Tracker
The tracker runs after every tool execution and updates enforcement state based on what happened.

6.1 Design Philosophy
Fail-open: If tracker crashes, tool execution is NOT blocked (observability, not security)
Merge-based state updates: Uses update_state() with a merge function that preserves concurrent changes from other hook invocations
State mutation is the tracker's responsibility — gates are read-only
6.2 What It Tracks
Tool    What's Tracked    State Field    Used By
Read    File paths read    files_read    Gate 1
Bash (VPS pull)    scp/rsync FROM VPS    last_vps_pull    Gate 7
Bash (VPS backup)    cp .bak commands    last_vps_backup    Gate 20
Bash (SSH)    SSH commands    last_vps_ssh    Gate 22
Bash (tests)    pytest/compile runs    tests_run_this_session    Gate 11
Bash (deploy)    scp/rsync TO VPS    Cleanup of .vps_current files    —
Task    Spawn timestamps    task_spawns    Gate 24
Edit/Write    Edited .py files    pending_verification_files    Gate 13
Read/Grep/Bash(test)    Verification of edited files    Clears pending_verification_files, sets fix_needs_akira_save    Gate 13/19
6.3 Merge Strategy
The tracker uses a snapshot-and-merge approach to handle concurrent state modifications:

# Build snapshot of fields tracker changed
snapshot = {k: state[k] for k in tracked_fields if k in state}

def _merge_tracker(disk_state, _snap=snapshot):
    for key, val in _snap.items():
        if isinstance(val, list) and isinstance(disk_state.get(key), list):
            if len(val) == 0:
                disk_state[key] = []  # Intentional clear (e.g., test run)
                continue
            # Merge: add items from snapshot not already on disk
            existing = set(str(x) for x in disk_state[key])
            for item in val:
                if str(item) not in existing:
                    disk_state[key].append(item)
        elif isinstance(val, dict) and isinstance(disk_state.get(key), dict):
            disk_state[key].update(val)  # Union of entries
        else:
            disk_state[key] = val

update_state(_merge_tracker)
7. Memory System (AKIRA MCP Server)
7.1 Overview
AKIRA's memory is powered by a ChromaDB vector database exposed via the MCP (Model Context Protocol) standard. The MCP server runs as a stdio subprocess managed by Claude Code.

7.2 Architecture
Claude Code  ←→  MCP Server (stdio)  ←→  ChromaDB (HTTP, port 8111)
                  (Python process)         (Separate process, pre-started)
Why separate processes: On Windows, Claude Code's process tree kills child subprocesses when compacting context. ChromaDB must be started independently BEFORE Claude Code launches.

7.4 Quality Controls
Ingestion filter (pre-save): 6 regex rules block noise before it enters the database. Prevents low-quality content from polluting search results.

Content-hash deduplication (S638): 2-layer dedup at the MCP server level — MD5 content hash (exact match) + embedding cosine similarity (semantic match). Prevents duplicate memories from entering the database even if phrased differently.

Tier system (post-save classification):

Tier 1: High-quality (bug fixes with root cause, architectural decisions, deployment records)
Tier 2: Standard quality (session summaries, general notes)
Tier 3: Low quality (raw dumps, temporary context)
Taxonomy: 24 historical labels unified to 9 canonical categories: BUG FIX, PLAN, DECISION, DEPLOY, DISCOVERY, CODE, TRADING, SESSION, MEMORY

Project tagging: Memories tagged by project scope: trading, infrastructure, akira, tools, general.

7.5 Database Statistics
Total memories: 20,950+
Code patterns: 18
Collections: memories (main), learned_code (patterns)
Storage: ~250MB on disk
Embedding model: BGE-M3 (1024-dim, GPU-accelerated) — upgraded from ChromaDB default in S634 rebuild
8. Boot Sequence
8.1 SessionStart Hook (boot_v5.py)
When Claude Code starts a new session, the boot script:

Checks system integrity:

Verifies all 11 gate module files exist and are non-empty
Verifies all 13 hook events are configured in settings.json
Verifies all 6 agent definition files exist
Verifies all 3 rule files exist
Cross-validates LIVE_STATE.json internal consistency
Loads LIVE_STATE.json for VPS and trading status

Detects data staleness: Computes how old the LIVE_STATE data is (hours/days since last_updated)

Reads HANDOFF.md: Extracts session number and accomplishment summary from previous session

Resets enforcement state: Clears all session-specific fields, increments session number

Checks for anomalies: Validates numbers against defined bounds:

Win rate: 40-90%
Daily PnL: -$100 to +$100
Daily trades: 0-50
PID: >1000
Memory: 20-500MB
Values outside bounds are tagged with [ANOMALY]
Checks for high-risk times: Friday (29% frustration rate) or late night 11PM-3AM (32.2% frustration at 2AM)

Outputs dashboard: ANSI-colored box-style dashboard to stdout:

8.2 LLM Boot Protocol
After the SessionStart hook fires, the LLM executes a specific 3-call parallel boot:

# All 3 in ONE message (parallel execution):
Task(subagent_type="boot-gatherer", model="opus")  # AKIRA recent activity (48h)
Read("HANDOFF.md")                                   # Previous session handoff
Read("LIVE_STATE.json")                              # Current system state
The LLM then renders a dashboard and says "Ready." with no additional chit-chat.

10. Agent Teams
10.1 Overview
Agent Teams enable coordinated multi-agent workflows with a team lead and named teammates. Teams use shared task lists for work coordination.

10.2 Team Lifecycle
1. TeamCreate(name, description)
   → Creates team config at ~/.claude/teams/{name}/config.json
   → Creates task list at ~/.claude/tasks/{name}/

2. TaskCreate(subject, description)  × N
   → Creates tasks in the team's task list

3. Task(name="agent-name", team_name="team", subagent_type="...", model="opus")  × N
   → Spawns named teammates that join the team
   → Each teammate can read the task list, claim tasks, complete them

4. TaskUpdate(taskId, addBlockedBy=[...])
   → Sets up dependency chains between tasks

5. Teammates work → claim tasks → complete them → go idle
   → SendMessage for inter-agent communication
   → Team lead coordinates via messages and task assignments

6. SendMessage(type="shutdown_request", recipient="agent-name")
   → Graceful shutdown of teammates when work is done

7. TeamDelete()
   → Cleans up team and task directories
10.3 Advanced Features
Plan Mode Required: Teammates can be forced to plan before implementing:

Task(name="deployer", mode="plan", team_name="audit-team")
# Teammate must call ExitPlanMode → team lead approves → implementation begins
Max Turns: Prevent runaway agents:

Task(name="researcher", max_turns=20, team_name="audit-team")
Task Dependencies: Pipeline workflows with auto-unblocking:

TaskUpdate(taskId="3", addBlockedBy=["1", "2"])
# Task 3 cannot start until tasks 1 and 2 are completed
10.4 Platform Constraints
Sibling tool call error cascade: Claude Code batches parallel tool calls atomically. If ONE fails (e.g., a gate blocks a Task spawn), ALL sibling calls in the same batch die. Mitigation: avoid parallel calls when gate blocks are likely.
In-process mode: On Windows, teammates run in-process (not in separate terminal panes). Split-pane requires tmux/iTerm2 (Unix only).
Optimal team size: 3-6 teammates. One team per session. No nested teams.

12.2 Enforcer Code Rules (Critical)
When editing any hook or gate file, these rules are injected:

Gate Contract:
1. Export check(tool_name, tool_input, state) -> (bool, str)
2. Return (True, "") to allow, (False, "message") to block
3. Be READ-ONLY — gates NEVER modify state.json
4. Be FAIL-CLOSED — exceptions cause exit(2), not bypass

Critical Rules:
- NEVER add state fields without updating get_default_state() AND reset_session_state()
- Exit codes: 0 = allow, 2 = block (NOT 1 — that means hook error)
- Messages go to stderr (not stdout)
- Plan-approved has 60-min TTL

14. Auto-Approve System
14.1 Purpose
Reduces friction for safe, repetitive commands. Instead of the human clicking "Allow" for every git status or SSH check, safe commands are auto-approved.

14.2 How It Works
PermissionRequest event → auto_approve.py

1. Parse the Bash command from stdin JSON
2. Split chained commands (&&, ||, |, ;)
3. For EACH part:
   a. Check against DANGEROUS_PATTERNS (rm -rf, DROP TABLE, etc.) → DENY ALL
   b. Check against SAFE_PATTERNS (SSH to known hosts, git status, ls, etc.)
   c. ALL parts must be safe, ANY dangerous part = deny
4. Recursively check subshells: $(...), `...`, bash -c "..."
5. If all safe: print "approve" to stdout
6. If any unsafe: exit silently (show normal permission dialog)
14.3 Safe Patterns
SAFE_PATTERNS = [
    # VPS SSH (dynamically built from constants.py VPS_HOSTS)
    "ssh root@<VPS_IP>",
    "scp root@<VPS_IP>",

    # Git (read-only)
    "git status", "git log", "git diff", "git branch",

    # File inspection
    "ls ", "cat ", "grep ", "head ", "tail ",

    # Testing
    "python -m pytest",

    # Service inspection
    "systemctl status", "journalctl",
]

16.3 Testing Approach
Tests verify both positive (tool should be allowed) and negative (tool should be blocked) cases:

def test_no_destroy_rm_rf():
    result = run_enforcer(tool_name="Bash", tool_input={"command": "rm -rf /"})
    assert result.returncode == 2
    assert "GATE 6" in result.stderr

# Example: Test that Gate 6 allows grep containing rm -rf (searching, not executing)
def test_no_destroy_allows_grep():
    result = run_enforcer(tool_name="Bash", tool_input={"command": "grep 'rm -rf' file.log"})
    assert result.returncode == 0
17. Design Principles
17.1 Fail-Closed vs Fail-Open
Component    Failure Mode    Rationale
Gates (PreToolUse)    Fail-CLOSED    Broken gate = block action. Security boundary.
Tracker (PostToolUse)    Fail-OPEN    Broken tracker = don't block work. Observability only.
Audit logging    Fail-SILENT    Never raise. Never block.
Auto-approve    Fail-to-dialog    If unsure, show normal permission dialog.
Boot integrity    Warn-only    Missing gates = warning in dashboard, not crash.
17.2 TOCTOU Awareness
Time-of-Check-Time-of-Use race conditions are acknowledged and handled:

State management: Atomic update_state() with file locking for all read-modify-write operations
Gate checks: State may change between gate check and tool execution. Gates are advisory (behavioral guidance), not security barriers against a hostile LLM.
File existence: os.path.exists() checks in Gate 1 are best-effort. Race condition accepted — the gate prevents the common mistake (editing without reading), not adversarial exploitation.
17.3 Single Source of Truth
Constants are defined ONCE in shared/constants.py and imported everywhere:

VPS host IPs
Pattern lists (deploy, pull, test, backup)
Freshness timeouts
Number bounds
Log rotation sizes
17.4 Idempotent State Resets
Session state reset is idempotent — calling reset_session_state() multiple times produces the same result. The session_reset_done flag prevents redundant resets within a session.

17.5 Gradual Trust Escalation
The system implements trust levels:

EDIT LOCKED (session start)
  → Query AKIRA (unlocks basic edits)
    → Enter Plan Mode (unlocks cursed/trading files)
      → Save plan to AKIRA (unlocks plan exit)
        → Exit Plan Mode (plan_approved=True, 60-min TTL)
          → Edit → Verify → Save to AKIRA (full cycle)
18. Failure Modes & Mitigations
18.1 Known Platform Constraints
Issue    Impact    Mitigation
Sibling tool call cascade    If one parallel tool call fails (gate block), ALL siblings in the batch die    Avoid parallel calls when gate blocks are likely. Put risky calls in separate batches.
Context window compression    On compact, conversation history is summarized (lossy)    PreCompact hook backs up state files and injects critical context summary into compacted output
MCP subprocess lifetime (Windows)    Windows kills child subprocesses when parent compacts    Start ChromaDB as independent process before Claude Code. Detection function checks if already running.
Hook timeout    Hooks that exceed their timeout are killed    All hooks have conservative timeouts (1-15s). Critical hooks (enforcer) have 5s budget for all 13 gates.
18.2 Recovery Mechanisms
Scenario    Recovery
state.json corrupted    load_state() catches JSONDecodeError, resets to defaults
Lock file stuck (crashed hook)    Stale lock detection: >30s old → break lock and retry
AKIRA DB unavailable    Gates that check AKIRA freshness will block edits until DB is reachable. Manual override: set akira_last_queried in state.json.
Gate module file deleted    Boot integrity check warns. Enforcer's from gates import GATE_ORDER fails → fail-closed (blocks ALL tool calls).
Session state drift    /wrap-up cross-validates all 3 state files (HANDOFF.md, LIVE_STATE.json, MEMORY.md). Boot runs check_state_consistency().
18.3 Accepted Risks
These are known issues that have been analyzed and accepted:

parse_utc_ts() timezone coercion: naive timestamps treated as UTC (matches actual usage)
Grep/xargs pipe bypass in Gate 6: unlikely the LLM generates grep | xargs rm (pipe chains are complex)
PostToolUse receives tool_input, not output: cannot verify SSH success, only that SSH was attempted
Duplicate logging boilerplate in teammate_idle/task_completed: cosmetic, not functional
Force parameter in AKIRA save has no audit trail: design decision for emergency saves
