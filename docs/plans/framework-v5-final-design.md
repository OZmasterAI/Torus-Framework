# Framework v5: Final Design

> **Session:** 529 | **Date:** 2026-03-25
> **Inputs:** 529 sessions experience, OpenClaw comparison, DeepSeek Engram paper, 4 research agents (web + 28 local research files), complete Torus audit (21 gates, 34 hooks, 73 modules, 46 skills, 5 MCP servers)
> **Priority:** Quality → Consistency → Reliability → Efficiency
> **Philosophy:** Files are truth. Wikilinks are the graph. Indexes are disposable. Access is hierarchical. Mechanical enforcement over behavioral instruction.

# 1. MEMORY SYSTEM

## 1.1 Architecture: 6 Layers

```
┌──────────┬──────────────┬──────────────┬──────────────┬────────────────┐
│ L0       │ L1           │ L2           │ L3           │ L4             │
│ INDEX    │ SESSION      │ KNOWLEDGE    │ SEARCH       │ INTELLIGENCE   │
│ routing  │ ephemeral    │ PARA files   │ FTS5+Vector  │ quality+graph  │
│ preloaded│ NOT truth    │ IS truth     │ disposable   │ adaptive       │
├──────────┼──────────────┼──────────────┼──────────────┼────────────────┤
│ CLAUDE.md│ state.json   │ ~/knowledge/ │ index.db     │ KG + quality   │
│ _index.md│ working-ctx  │ with [[wiki]]│ vectors/     │ + decay + LTP  │
└──────────┴──────────────┴──────────────┴──────────────┴────────────────┘
  + L5: DAILY NOTES (areas/daily/YYYY-MM-DD.md)
  + L6: CAUSAL CHAINS / TIPS (resources/fix-history/)
```

## 1.2 Layer 0: Index / Routing

- `~/knowledge/_index.md` — routing file, common searches, PARA folder map
- `~/.claude/CLAUDE.md` — behavioral rules, workflow loop, skill triggers
- `~/.claude/rules/` — gate contract, safety, memory, framework rules
- Conditional prompt assembly — load only fragments relevant to current project/mode (inspired by Claude Code's 250-fragment system with template variables)

Source: OpenClaw MEMORY.md + AGENTS.md separation, DeepSeek Engram L0 preload concept, Claude Code conditional assembly

## 1.3 Layer 1: Session Memory (Ephemeral — NOT truth)

- `state.json` — session_count, project, what_was_done, next_steps, known_issues
- 9-section session template (Anthropic pattern): Current State, Task Spec, Files/Functions, Workflow, Errors/Corrections, Learnings, Key Results, Worklog
- `working-context.md` — two sections:
  - **Status** (auto-generated by tracker every PostToolUse): activity, errors, verification ratio, key files
  - **Summary** (Claude-written at threshold via /working-summary): goal, approach, progress, decisions, gotchas, next steps

**Context Preservation Lifecycle:**
```
CONTINUOUS:   tracker updates Status section every PostToolUse
AT THRESHOLD: gate blocks → Claude writes Summary → gate unblocks
PRE-COMPACT:  expand working-context.md with full error history + unresolved items
POST-COMPACT: re-inject working-context.md + _index.md
SESSION END:  append to daily note, preserve Summary for continuity
```

Source: Ours (working-memory + working-summary merged), OpenClaw ("session ≠ truth"), Continuous-Claude-v3 (independently converged on same pattern), Anthropic session memory template

## 1.4 Layer 2: Durable Knowledge (PARA Files = Canonical Truth)

`~/knowledge/` organized as PARA with wikilinks:
- `projects/` — active projects (torus-framework.md, go-sdk-agent.md)
- `areas/` — ongoing domains (memory-systems.md, testing-patterns.md)
- `areas/daily/` — auto-generated daily rollup (YYYY-MM-DD.md)
- `resources/` — reference material + fix-history/
- `archives/` — superseded knowledge (not deleted, moved with superseded_by link)

Every file has YAML frontmatter:
```yaml
tags: [area, active]
quality: 0.85
created: 2026-03-25
last_accessed: 2026-03-25
access_count: 12
type: reference          # reference | working
trigger_condition: ""    # when to apply this knowledge (IBM paper)
negative_example: ""     # what NOT to do (IBM paper, +149% on hard tasks)
tip_type: ""             # strategy | recovery | optimization (IBM taxonomy)
```

Every file has `[[wikilinks]]` to related topics → Obsidian Graph View renders the knowledge graph automatically.

Source: OpenClaw PARA, vault mirroring (Gigabrain), wikilinks (new), Obsidian as interface (new), IBM trajectory memory (tip schema), AutoAgent (dual-representation)

## 1.5 Layer 3: Search (Disposable Indexes)

Engram-inspired cascade — cheapest first, expensive only when needed:

```
search_knowledge(query)
  L0: _index.md → known topic name? → return file path          (~80%)
  L1: Glob ~/knowledge/**/*.md → filename match? → return file   (~10%)
  L2: SQLite FTS5 keyword search → return matched snippets        (~8%)
  L3: Vector semantic search (nomic-v2-moe 768-dim) → nearest     (~2%)

  Adaptive hybrid ratio: 85% cosine / 15% BM25 default, 70/30 for short queries
  Side effect: knowledge_graph.db.strengthen_coretrieval()
```

Both indexes are disposable. Delete → rebuild from `~/knowledge/` files in ~2 min.

Source: DeepSeek Engram hierarchy, ENGRAM paper (typed retrieval + dense search beats full context by 15 pts using 1% tokens), Memori (adaptive hybrid ratio), OpenClaw ("DB doesn't replace PARA")

## 1.6 Layer 4: Intelligence

### Quality + Dedup
- Quality scoring (0-1): identifier density, causal language, outcome presence, structure
- Consensus validator: multi-signal validation on memory saves and edits (duplicate/conflict/novel)
- Dedup: cosine < 0.12 hard block, 0.12-0.20 soft tag

### Decay + LTP
- Hybrid decay: exponential (half-life 15 days) → power-law (t^-0.5 tail, memories never fully vanish)
- LTP tracker: 4 levels (none/burst/weekly/full) modulate decay rate
- Summarize-and-replace archival: don't just decay — summarize before archiving

### Knowledge Graph
- `knowledge_graph.py` (~560 lines) + `knowledge_graph.db` (4,929 entities, 6,205 edges)
- Hebbian co-retrieval strengthening on every search
- PMI weighting, spreading activation, cluster detection
- Graph MUST be traversed at query time (Memori anti-pattern: stored but never queried = dead weight)
- Feeds wikilink recommendations when writing new knowledge files
- Dual representation: wikilinks (human-visible) + SQLite (computational)

### Consolidation + Maintenance
- Merge candidates (Jaccard word overlap >= 0.5)
- Promotion (relevance > 0.7, retrieval >= 10)
- Archive (relevance < 0.15, no LTP protection)
- Health analysis, stale scan, cleanup recommendations
- Gap detection: identify what memory lacks (MemOS pattern)

### Dream Consolidation Agent (NEW)
Scheduled periodic maintenance (Anthropic built this into Claude Code):
1. **Orient**: scan knowledge directory, read _index.md
2. **Gather signal**: scan recent daily notes, grep session transcripts
3. **Consolidate**: merge related facts into topic files, convert relative dates to absolute, delete contradicted facts
4. **Prune**: keep _index.md under N lines, remove stale pointers

Source: Ours (quality, decay, LTP, KG), IBM paper (tip schema, gap detection), AutoAgent (tool profiles, episodic clustering), Anthropic (dream consolidation), Memori (anti-pattern warning), MemOS (gap detection)

## 1.7 Layer 5: Daily Notes

`~/knowledge/areas/daily/YYYY-MM-DD.md` — one file per day, append sessions:
```markdown
## Session 529
- Compared OpenClaw memory stack vs Torus
- Designed clean-sheet framework

```

Source: OpenClaw daily files, ours (auto-generation)

## 1.8 Layer 6: Causal Chains / Tips

Upgraded from simple fix_outcomes to IBM trajectory tip store:

`~/knowledge/resources/fix-history/{fingerprint}.md`:
```markdown
---
error_pattern: "ModuleNotFoundError: No module named 'shared.xyz'"
tip_type: recovery          # strategy | recovery | optimization
trigger_condition: "importing from shared/ in a new framework file"
negative_example: "Don't use sys.path.insert — breaks in different working directories"
causal_level: root_cause    # immediate | proximate | root_cause
confidence: 0.82
attempts: 3
source_trajectory: session-529
---
## What Worked
Use relative import: `from .xyz import func`

## What Didn't Work
### 1. sys.path.insert (FAILED, 2 attempts)
Path was wrong for ramdisk execution context
```

MCP tools: `record_attempt`, `record_outcome`, `query_fix_history` — same API, richer storage.

Source: Ours (causal chains) + IBM paper (tip taxonomy, trigger conditions, negative examples, multi-level attribution, +149% on hard tasks)

## 1.9 Write Path

```
remember_this(content, tags)
  1. Quality score → YAML frontmatter
  2. Consensus check (duplicate? conflict? novel?)
  3. Classify: type (reference/working), tip_type, trigger_condition, negative_example
  4. Dual-representation: store raw content + generate compressed summary
  5. Determine PARA location from tags
  6. Write/update ~/knowledge/{location}/topic.md
  7. Query knowledge_graph.db for related entities
  8. Auto-insert [[wikilinks]] to connected concepts
  9. Index into FTS5 (L2)
  10. Index into vector store (L3)
  11. Update knowledge_graph.db edges (Hebbian)
```

# 2. GATE SYSTEM

## 2.1 Risk-Based Routing (NOT all gates every call)

```
Tool call arrives
  → Classify risk: READ (low) | WRITE (medium) | DESTROY (high) | DEPLOY (critical)
  → Route to appropriate gate set:

  READ tools (Read, Glob, Grep, WebSearch):
    T1 only (gates 1-2) — fast path, <10ms

  WRITE tools (Edit, Write, NotebookEdit):
    T1 + T2 (gates 1-9) — full quality enforcement

  DESTROY tools (Bash with rm/drop/reset):
    T1 + T2 + T3 (all gates) — maximum scrutiny

  DEPLOY tools (Bash with scp/rsync/docker push):
    T1 + T2 + T3 + deploy-safety check
```

Source: Ruflo 3-tier routing, industry 3-layer consensus (rules <10ms, ML 50-200ms, LLM 300-2000ms), Datadog risk-based routing guide

## 2.2 The 15 Gates

### T1 — Safety (fail-closed)

| # | Gate | Triggers On | What |
|---|---|---|---|
| 1 | **Read-before-edit** | Edit, Write | Block unless file was Read first |
| 2 | **No-destroy** | Bash | Block rm -rf, DROP TABLE, force push, reset --hard, deploy without tests |

### T2 — Quality (fail-open)

| # | Gate | Triggers On | What |
|---|---|---|---|
| 3 | **Memory-first** | Edit, Write, Task | Block if memory not queried in 5 min |
| 4 | **Critical-file** | Edit, Write | High-risk files need memory query first |
| 5 | **Code-quality** | Edit, Write | Secret detection, debug artifacts, conventions |
| 6 | **Injection-defense** | WebFetch, WebSearch | Prompt injection scanning (6 attack categories) |
| 7 | **Test-required** | Edit, Write | Block code writes without tests (TDD) |
| 8 | **Proof-before-fixed** | Edit, Write | Block further edits until previous verified |
| 9 | **Save-fix** | Edit, Write | Block when verified fixes not saved to memory |
| 10 | **Causal-chain** | Edit, Write | Block after test failure until fix history queried |

### T3 — Advisory/Operational (fail-open)

| # | Gate | Triggers On | What |
|---|---|---|---|
| 11 | **Strategy-ban** | Edit, Write | Block when fix strategy has 3+ failures |
| 12 | **Rate-limit** | All | Block at >60 calls/min, warn at >40 |
| 13 | **Context-preservation** | Edit, Write, Bash, Task | Block at threshold until summary written |
| 14 | **Tool-profiles** | All profiled | Warn on known failure patterns (adaptive, learns) |
| 15 | **Canary** | All | Passive telemetry: bursts, repeats, anomalies |

### Gate Infrastructure
- Single `enforcer.py` — all gates in one file, direct execution
- `gate_result.py` — GateResult dataclass (40+ imports depend on it)
- Tiers: T1 fail-closed (exceptions block), T2+ fail-open (exceptions warn)
- Circuit breakers per gate — fault isolation (gate_06 had 41 crashes in production)
- Audit log — JSONL trail of all gate decisions
- Risk-based routing — READ gets fast path, WRITE/DESTROY gets full stack

### Future Gate Capabilities (Phase 3+)
- Adaptive thresholds learning from outcomes (AutoAgent: +6.6pp from online adaptation)
- Per-agent trust accumulation with privilege tiers (Ruflo pattern)
- Plan-level pre-execution analysis for multi-step sequences (Safiron/Pre-Exec Bench)
- Graph-based test impact analysis replacing file-name matching (TDAD, 70% regression reduction)

# 7. FEEDBACK LOOPS (NEW SECTION)

Three feedback loops that make the framework learn:

## 7.1 PostToolUse Feedback (tracker.py)
Every tool call → extract errors, track verification, update tool profiles, update working-context.md. **Blue ocean — no other framework has this.**

## 7.2 LLM-as-Judge Self-Evaluation (Phase 2+)
After task completion → automated quality scoring → store in memory → feed into future behavior. (Reflexion framework, OpenAI cookbook)

## 7.3 Insights/Friction Analysis (Phase 3+)
Periodically extract session facets → identify recurring friction → generate improvement suggestions → the framework improves itself. (Anthropic v2.1.72+ pattern)

# 8. SAFETY (NEW SECTION)

## 8.1 Shadow Git Checkpoints (NEW)
Before any file mutation, snapshot working directory to separate `~/.framework/checkpoints/` via isolated GIT_DIR. 50-checkpoint rolling limit. Undo-the-undo. Safety net independent of user's git. (Hermes pattern)

## 8.2 Autonomous Mode Safety
Security monitor evaluates agent actions against BLOCK/ALLOW rules:
- Composite action evaluation (multi-step intent)
- Written file execution tracking
- "Questions are not consent", "Silence is not consent"
(Anthropic Security Monitor pattern, ~5,700 tokens)

## 8.3 External Agent Validation
When external agents (KiloCode, OpenCode) return results:
- Post-merge gate validates diffs before committing to main tree
- Git worktree isolation mandatory for file-editing sub-agents
(External delegation research)

7. **Dream consolidation** + **self-improving skills** + **insights system** as a three-part self-improvement loop.
