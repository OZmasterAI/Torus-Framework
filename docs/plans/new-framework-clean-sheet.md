# Design: Clean-Sheet Framework & Memory System (v4)

> **Status:** SELECTED — Design A (Quality-First) | **Session:** 529 | **Date:** 2026-03-25
> **Audit:** Complete — 21 gates, 34 hooks, 73 shared modules, 46 skills, 5 MCP servers reviewed
> **Priority:** Quality, Consistency, Reliability first. Efficiency second.

## Problem

After 529 sessions building the Torus Framework, we have deep experience with what works and what doesn't in agent memory, gate enforcement, session lifecycle, and knowledge management. A comparison with OpenClaw's memory stack revealed a fundamentally different philosophy — files as durable truth vs our vector-DB-as-truth — that's better for human visibility and resilience.

This is a new framework built from everything we've learned. Not a refactor, not an optimization — a clean-sheet design informed by 529 sessions of real usage.

## Design Priority

**Quality → Consistency → Reliability → then Efficiency.**

The framework exists to maximize the quality, consistency, and reliability of:
1. **Memory** — what we know and how well we recall it
2. **Output** — what we produce for the user
3. **Code** — what we write and how well it works
4. **Learning** — how well we improve from experience
5. **Multi-agent** — how well parallel work coordinates
6. **The framework itself** — how robust and trustworthy the system is

Code length is not a constraint. Token usage is not a constraint. RAM is not a constraint. These are secondary concerns addressed in Design B (Lean Variant) at the end.

## Core Philosophy

**Files are truth. Wikilinks are the graph. Indexes are disposable. Access is hierarchical. Mechanical enforcement over behavioral instruction.**

# PART 1: MEMORY SYSTEM

## Memory Architecture: 6 Layers

```
┌──────────┬──────────────┬──────────────┬──────────────┬────────────────┐
│ L0       │ L1           │ L2           │ L3           │ L4             │
│ INDEX    │ SESSION      │ KNOWLEDGE    │ SEARCH       │ INTELLIGENCE   │
│ routing  │ ephemeral    │ PARA files   │ FTS5+Vector  │ quality+graph  │
│ preloaded│ NOT truth    │ IS truth     │ disposable   │ adaptive       │
├──────────┼──────────────┼──────────────┼──────────────┼────────────────┤
│ CLAUDE.md│ state.json   │ ~/knowledge/ │ index.db     │ KG + quality   │
│ _index.md│ working-ctx  │ with [[wiki]]│ vectors/     │ + decay + LTP  │
└──────────┴──────────────┴──────────────┴──────────────┴────────────────┘
  + L5: DAILY NOTES (areas/daily/YYYY-MM-DD.md)
  + L6: CAUSAL CHAINS (resources/fix-history/)
```

#### Context Preservation System

Prevents context rot across compaction boundaries.

**`working-context.md` — two sections, two writers:**

```markdown
# Working Context
## Status (auto-generated by tracker every PostToolUse)
Active: editing enforcer.py
Errors: ImportError in shared/quality.py (2x)
Verified: 3/5 edits tested
Key files: enforcer.py, memory_server.py, test_enforcer.py
Unresolved: quality.py import path

## Summary (Claude-written at threshold via /working-summary)
Goal: Implement Phase 1 of clean-sheet framework
Approach: TDD — test files first, then implementation
Progress: enforcer.py done, session.py in progress
Key decisions: nomic embeddings kept, quality-first design
Gotchas: Gate 23 blocks writes without tests — intentional
Next: memory_server.py with FTS5
```

**Lifecycle:**
```
CONTINUOUS: tracker updates Status section every PostToolUse
AT THRESHOLD: gate blocks → Claude writes Summary via /working-summary
PRE-COMPACT: expand working-context.md with full error history + unresolved items
POST-COMPACT: re-inject working-context.md + _index.md into new context
SESSION END: append key items to daily note, preserve Summary for continuity
```

### Layer 2: Durable Knowledge (PARA Files = Canonical Truth)

| Component | Source | Decision |
|---|---|---|
| `~/knowledge/{projects,areas,resources,archives}/` | OpenClaw PARA | Human-browseable, git-tracked, survives any tool change |
| Vault mirroring (`remember_this` → write file) | OpenClaw Gigabrain | Every captured fact → readable markdown file |
| Wikilinks `[[related topic]]` | New | Obsidian renders graph. Neither system had visible edges |
| Obsidian as browse/graph interface | New | `~/knowledge/` IS an Obsidian vault |
| YAML frontmatter | Ours | tags, quality, last_accessed, access_count, type, created |
| "Supersede, don't delete" | OpenClaw | Move to `archives/` with `superseded_by:` link |
| "Never store secrets" | OpenClaw | Enforced by code-quality gate |

### Layer 3: Search (Disposable Indexes)

| Component | Source | Decision |
|---|---|---|
| Engram L0→L3 cascade | DeepSeek paper | Cheapest lookup first, expensive only when needed |
| SQLite FTS5 (L2) | New | Indexes all ~/knowledge/**/*.md. Disposable — rebuild in ~5s. |
| Vector semantic (L3) | Ours | nomic-embed-text-v2-moe (768-dim). Proven quality. No downgrade. |
| "DB doesn't replace PARA" | OpenClaw | Both indexes disposable. Delete → rebuild from files. |

**Read path:**
```
search_knowledge(query)
  L0: _index.md → known topic name? → return file path
  L1: Glob ~/knowledge/**/*.md → filename match? → return file
  L2: SQLite FTS5 keyword search → return matched snippets
  L3: Vector semantic search → return nearest matches
  Side effect: knowledge_graph.db.strengthen_coretrieval()
```

### Layer 4: Intelligence (Quality + Graph + Decay + Maintenance)

| Component | Source | For QCR |
|---|---|---|
| Quality scoring (0-1) | Ours | **Memory quality** — prevents low-value entries |
| Dedup (cosine + FTS5) | Ours | **Memory consistency** — prevents duplicate knowledge |
| Decay (hybrid exp→power-law) | Ours | **Memory quality** — stale memories fade, relevant ones persist |
| LTP tracker (none/burst/weekly/full) | Ours | **Memory reliability** — frequently-used knowledge gets protected |
| Knowledge graph (Hebbian, PMI) | Ours | **Memory quality** — co-retrieval strengthening, concept clustering |
| Error normalizer (FNV1a) | Ours | **Consistency** — deterministic IDs for dedup and causal chains |
| Consolidation (merge/promote/archive) | Ours | **Memory consistency** — periodic cleanup of redundant/stale entries |
| Maintenance (health analysis, stale scan) | Ours | **Memory reliability** — detect degradation before it affects quality |
| Consensus validator | Ours | **Reliability** — multi-signal validation on memory saves and edits |
| Classification (reference/working/ephemeral) | Ours | **Consistency** — memories have clear lifecycle expectations |

**Write path:**
```
remember_this(content, tags)
  1. Quality score → YAML frontmatter
  2. Consensus check (duplicate? conflict? novel?)
  3. Determine PARA location from tags
  4. Write/update ~/knowledge/{location}/topic.md
  5. Query knowledge_graph.db for related entities
  6. Auto-insert [[wikilinks]] to connected concepts
  7. Index into FTS5 (L2)
  8. Index into vector store (L3)
  9. Update knowledge_graph.db edges (Hebbian)
```

# PART 2: GATE SYSTEM

## 15 Gates

Quality-first means keeping every gate that improves output quality, consistency, or reliability — even if it costs more lines or time.

### T1 — Safety (fail-closed: exceptions block)

| # | Gate | What | QCR Impact |
|---|---|---|---|
| 1 | **Read-before-edit** | Block Edit/Write unless file was Read first | **Consistency** — prevents blind edits |
| 2 | **No-destroy** | Block rm -rf, DROP TABLE, force push, reset --hard, deploy without tests | **Reliability** — prevents catastrophic damage |

### T2 — Quality (fail-open: exceptions warn)

| # | Gate | What | QCR Impact |
|---|---|---|---|
| 3 | **Memory-first** | Block work until knowledge queried (5 min window) | **Quality** — ensures informed decisions |
| 4 | **Critical-file** | High-risk files need memory query first | **Reliability** — protects core framework files |
| 5 | **Code-quality** | Secret detection, debug artifacts, convention violations | **Quality** — prevents security leaks and code smell |
| 6 | **Injection-defense** | Prompt injection scanning on web/MCP results | **Reliability** — security against external content |
| 7 | **Test-required** | Block code writes without corresponding tests | **Quality** — TDD enforcement |
| 8 | **Proof-before-fixed** | Block further edits until previous edit verified | **Consistency** — forces verification before moving on |
| 9 | **Save-fix** | Block edits when verified fixes not saved to memory | **Reliability** — ensures fixes are persisted |
| 10 | **Causal-chain** | Block edit after test failure until fix history queried | **Learning quality** — ensures past solutions are checked |

### T3 — Advisory/Operational (fail-open)

| # | Gate | What | QCR Impact |
|---|---|---|---|
| 11 | **Strategy-ban** | Block when fix strategy has 3+ failures | **Consistency** — prevents repeat mistakes |
| 12 | **Rate-limit** | Block at >60 tool calls/min, warn at >40 | **Reliability** — prevents runaway loops |
| 13 | **Context-preservation** | Block tools at threshold until summary written | **Reliability** — context rot prevention |
| 14 | **Tool-profiles** | Warn on known failure patterns for current tool | **Quality** — learns from past failures, adapts |
| 15 | **Canary** | Passive telemetry: call bursts, repeat sequences, new tools | **Consistency** — early warning for anomalous behavior |

### Gates Not Included (with rationale)

| Old Gate | Why Not |
|---|---|
| 10 (model-enforcement) | Multi-agent only. Add when multi-agent work begins. |
| 13 (workspace-isolation) | Multi-agent only. Add when multi-agent work begins. |
| 14 (confidence-check) | High false-positive rate. Quality cost > quality benefit. |
| 19 (hindsight) | Requires mentor scoring that's not proven to improve outcomes. |
| 20 (self-check) | Advisory questions already handled by behavioral rules in CLAUDE.md. |

### Gate Infrastructure

| Component | Decision | QCR Reason |
|---|---|---|
| `enforcer.py` (15 gates) | Direct execution, single file | **Reliability** — one file to debug |
| `gate_result.py` | GateResult dataclass | **Consistency** — standard result format |
| Tiers (T1/T2/T3) | T1 fail-closed, T2+ fail-open | **Reliability** — safety gates never skip |
| Enforcer daemon (optional) | Keep available for latency-critical sessions | **Quality** — faster gate checks = less user waiting |
| Circuit breakers | Keep per-gate | **Reliability** — one broken gate doesn't cascade |
| Audit log | JSONL trail of all gate decisions | **Reliability** — operational visibility |

# DESIGN B: LEAN VARIANT (Reference Only)

For contexts where efficiency matters more than maximum quality (e.g., lightweight deployments, resource-constrained environments):

- 10 gates (drop T2 quality gates 8-10, T3 advisory gates 14-15)
- Simplified tracker (no mentor signals, tool profiles, or consolidation)
- No circuit breakers, no enforcer daemon
- No consensus validator, no maintenance
- Minimal rules files
- Same memory system (PARA + FTS5 + vector — this is already efficient)
- Same TDD enforcement (AC + ACT + Tests — non-negotiable even in lean mode)

This variant trades quality enforcement for speed and simplicity. Use Design A unless there's a specific reason not to.
